# Page 48 — Sliding Window Median: Two Heaps + Lazy Deletion (Interview Version)

Yesterday’s stream median gave you `add(x)` + `median()` with two heaps. Interviewers often push one step further:

“Now return the median for every window of size `k`.”

This is harder because values must both **enter** and **leave** the structure. Heaps are great at insert/pop-top, but terrible at deleting an arbitrary old value. The interview-safe fix is **lazy deletion**.

## Core idea

Keep the same two heaps:
- `small` (max-heap via negatives) for lower half
- `large` (min-heap) for upper half

Add a hash map `delayed[val] = count` for numbers that should be removed later when they reach a heap top.

Also track logical sizes (`small_size`, `large_size`) separate from physical heap lengths.

## Invariants

1. Ordering: every active value in `small` `<=` every active value in `large`
2. Balance: `small_size == large_size` or `small_size == large_size + 1`
3. Tops are clean before reading median (prune delayed values)

If you can state these clearly, the interviewer sees control.

## Interview-grade Python skeleton

```python
import heapq
from collections import defaultdict

class DualHeap:
    def __init__(self, k):
        self.small, self.large = [], []
        self.delayed = defaultdict(int)
        self.small_size = self.large_size = 0
        self.k = k

    def prune(self, heap):
        while heap:
            x = -heap[0] if heap is self.small else heap[0]
            if self.delayed[x] == 0:
                break
            self.delayed[x] -= 1
            heapq.heappop(heap)

    def rebalance(self):
        if self.small_size > self.large_size + 1:
            heapq.heappush(self.large, -heapq.heappop(self.small))
            self.small_size -= 1; self.large_size += 1
            self.prune(self.small)
        elif self.small_size < self.large_size:
            heapq.heappush(self.small, -heapq.heappop(self.large))
            self.small_size += 1; self.large_size -= 1
            self.prune(self.large)
```

`add(x)`: push into correct heap, update logical size, rebalance.

`remove(x)`: increment `delayed[x]`, decrement logical size based on comparison with `-small[0]`, prune if `x` is at top, rebalance.

`median()`:
- odd `k`: `-small[0]`
- even `k`: `(-small[0] + large[0]) / 2`

## Complexity you should say out loud

- Each insertion/removal is `O(log k)` amortized
- Median query is `O(1)`
- Space is `O(k)`

## Common interviewer traps

- Forgetting logical sizes (heap length lies with delayed elements)
- Returning median without pruning
- Rebalancing before classifying the remove side

If you frame this as “two heaps, delayed deletes, strict invariants,” you turn a scary hard problem into a controlled systems design exercise.
