# Page 44 — Top K Frequent Elements: Pick the Right Tradeoff in 60 Seconds

Yesterday’s `merge k lists` page was about using a min-heap to repeatedly pull the next best candidate. Today we stay in the same neighborhood with another frequent interview prompt:

“Given an integer array and an integer `k`, return the `k` most frequent elements.”

The trap here is overengineering. In interviews, you should quickly show two valid approaches, then choose one based on constraints.

## Fast decision framing

If interviewer just wants a solid solution: use a **min-heap of size `k`**.

If they push for optimal linear-time behavior and value range is manageable: discuss **bucket sort by frequency**.

Lead with heap first because it is easy to implement correctly under pressure.

## Heap approach (interview-safe default)

1. Build frequency map with `Counter`.
2. Iterate `(num, freq)` pairs.
3. Push `(freq, num)` into min-heap.
4. If heap size exceeds `k`, pop once.
5. Return numbers from heap.

Why it works: heap always keeps the `k` highest frequencies seen so far; smallest among them sits at the root and gets evicted when a better candidate arrives.

```python
from collections import Counter
import heapq


def top_k_frequent(nums, k):
    freq = Counter(nums)
    heap = []  # (count, value)

    for num, count in freq.items():
        heapq.heappush(heap, (count, num))
        if len(heap) > k:
            heapq.heappop(heap)

    return [num for _, num in heap]
```

Complexity:
- Let `u` = number of unique values
- Time: `O(n + u log k)`
- Space: `O(u + k)`

In many real interviews, this is exactly what they want.

## Bucket sort follow-up (when asked)

Because max frequency is at most `n`, create buckets where index = frequency:
- `buckets[f]` stores all numbers appearing `f` times.
- Walk buckets from high to low until `k` numbers collected.

This yields near-linear time `O(n)`, but code is a bit longer and easier to bug if rushed.

## What to say out loud

“I’ll count frequencies first. Then I’ll maintain a min-heap of size `k` keyed by frequency so I never store more candidates than needed. That gives me `O(n + u log k)` and is clean to implement.”

If they ask “can you do better?” you can add: “Yes—bucket by frequency for `O(n)` time.”

## Common mistakes

1. Sorting all unique numbers by frequency (`O(u log u)`) without mentioning better options.
2. Using max-heap of all values (works, but extra memory/work).
3. Forgetting that answer order is usually arbitrary unless specified.
4. Mixing up `n` (array length) with `u` (unique count) in complexity.

Interview pattern recognition: whenever prompt says “top-k by metric,” think **frequency map + bounded heap** before anything else.
